# Incubator Criteria

This page describes the evidence reviewers expect for OSL Incubator Program
applications and lifecycle reviews. Reviewers may ask for more information when
a project has unusual context, but they should not waive the baseline criteria
without a documented OSL decision.

## Required criteria for all applications

Every application should provide evidence that the project:

- has a clear purpose aligned with open science, research, education,
  public-interest technology, open-source infrastructure, or another OSL value;
- uses an OSI-approved open-source license;
- includes a public `LICENSE` file;
- includes a public `CODE_OF_CONDUCT.md` or equivalent Code of Conduct with a
  reporting path;
- has at least one active and reachable maintainer;
- has a public repository or public proposal repository;
- has a public issue tracker, project board, discussion channel, or documented
  way to discuss work;
- has a public communication channel or contact path for users and contributors;
- has maintainers willing to communicate with OSL mentors and reviewers;
- can be maintained in public and can welcome contributors in a respectful
  environment;
- will acknowledge OSL status in `README.md` and public documentation if
  accepted.

## Recommended readiness criteria

These items are strongly recommended before acceptance and may become required
for later stage changes or graduation:

- `README.md` explains project purpose, users, and current status;
- `CONTRIBUTING.md` or contributor instructions exist;
- roadmap or milestones for the next 3 to 6 months exist;
- starter issues or planned contributor tasks exist;
- basic user/developer documentation exists;
- tests, validation examples, or a plan to add them exists;
- architecture, design, or technical overview exists where relevant;
- release, versioning, or publication plan exists where relevant;
- `SECURITY.md` or vulnerability reporting path exists where relevant;
- maintainer list, roles, or decision-making process is documented;
- evidence of early users, adopters, contributors, citations, deployments,
  pilots, or community interest exists where available;
- maintainers can explain what kind of help they need from OSL.

## PoC-specific criteria

PoC is for ideas, prototypes, or very early repositories that need validation
before full incubation. A PoC application should include:

- a problem statement;
- an intended user or beneficiary group;
- a proposal, prototype, or early repository;
- at least one reachable maintainer;
- a public repository or proposal repository;
- license and Code of Conduct links or a concrete plan to add them quickly;
- initial tasks, questions, or experiments;
- a 1-3 month exploration plan;
- the kind of OSL support requested.

## Incubation-specific criteria

Full incubation expects enough structure for sustained work. An incubation
application should include:

- active maintainer participation;
- a roadmap or milestones for the next 3-6 months;
- public issues, tasks, or project board entries;
- contribution instructions or a plan to add them;
- documentation status and improvement plan;
- test, validation, or quality plan;
- release or publication plan where relevant;
- mentor/reviewer communication expectations;
- scientific review path when relevant.

## Graduation criteria

Graduation review should find evidence that the project can operate with less
direct OSL incubation support:

- incubation milestones are complete or clearly superseded;
- maintainer list and roles are current;
- governance or decision-making process is documented;
- contributor onboarding process is documented;
- user/developer documentation is sufficient for current users;
- release, versioning, or publication process is documented where relevant;
- maintenance and roadmap plans exist;
- security reporting process exists where relevant;
- adoption, usefulness, deployment, citation, or community evidence is provided
  where available;
- project status acknowledgement will be updated after graduation;
- scientific review requirements have been met when applicable.

## Scientific software review criteria

For scientific Python projects in pyOpenSci scope, OSL graduation depends on
acceptance by pyOpenSci. For scientific R projects in rOpenSci scope, OSL
graduation depends on acceptance by rOpenSci. Projects outside those scopes
should agree with OSL on an equivalent review path during incubation, not at the
last minute before graduation.

Applicants should identify the likely review path early and keep evidence links
current in stage-change and graduation requests.

## GSoC and internship readiness criteria

Incubation can make a project eligible to prepare ideas for OSL programs when
available, but it does not guarantee participation. Projects interested in GSoC,
internships, grants, or similar limited programs should provide:

- scoped project ideas;
- active mentors and backup reviewers;
- public issues or tasks;
- expected outcomes;
- contributor prerequisites;
- communication channels;
- review cadence and availability;
- enough maintainer capacity to support contributors.

Contributors must not be promised acceptance, funding, slots, or employment.

## Reasons an application may be declined

OSL may decline, defer, or close an application when:

- an OSI-approved license is missing;
- the public `LICENSE` file is missing;
- a Code of Conduct is missing;
- no active or reachable maintainer is identified;
- the repository is private, archived, or unavailable;
- project purpose is unclear;
- the project is not aligned with OSL's mission;
- community behavior appears unsafe or hostile;
- ownership or license status is unclear;
- the project is too early even for PoC and lacks a concrete plan;
- maintainer capacity is not available;
- the application lacks evidence and the applicant does not respond;
- the project cannot support public collaboration.

## Reasons incubation may be paused or ended

OSL may pause incubation, mark a project at risk, remove a listing, or archive
an OSL-hosted repository after human review when:

- maintainers are unreachable;
- there is no progress against milestones and no pause plan;
- required license or Code of Conduct materials are removed;
- the project becomes closed-source;
- the project becomes unsafe or hostile to contributors;
- maintainers request pause or removal;
- the project is archived or abandoned;
- the project no longer aligns with OSL values.

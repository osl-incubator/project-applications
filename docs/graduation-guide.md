# Graduation Guide

Graduation means a project completed its OSL incubation milestones and can
operate with less direct incubation support. Graduation is a human review based
on evidence in a graduation request issue.

## When to request graduation

Maintainers should request graduation when the project has:

- completed or clearly superseded incubation milestones;
- stable maintainer participation;
- documented governance or decision-making practices;
- contributor onboarding process;
- sufficient user/developer documentation;
- release, versioning, or publication process where relevant;
- maintenance and security reporting paths where relevant;
- evidence of usefulness or adoption where available;
- scientific review acceptance or equivalent evidence when required;
- a plan for repository location and public status after graduation.

## Evidence to include

A graduation request should include links or explanations for:

- completed milestones;
- current maintainer list and roles;
- governance or decision-making process;
- contributor onboarding process;
- documentation;
- release/versioning process;
- maintenance plan;
- user, adoption, citation, deployment, or pilot evidence where available;
- security reporting process where relevant;
- future roadmap;
- desired repository location after graduation;
- pyOpenSci acceptance link for scientific Python projects in scope;
- rOpenSci acceptance link for scientific R projects in scope;
- equivalent review evidence for other scientific projects;
- README/docs status acknowledgement update plan.

## Public comment period

OSL may open a public comment period before graduation approval. The recommended
duration is 14 days. During the comment period:

- users, contributors, mentors, and reviewers may comment;
- maintainers may respond and update evidence;
- reviewers may request clarifications;
- OSL records a final decision after the period ends.

## Decision outcomes

Possible outcomes:

- approved graduation;
- needs more work;
- deferred for later review;
- declined for now.

If graduation is approved, OSL and maintainers should coordinate labels, public
listing updates, repository moves if needed, and acknowledgement updates.

## After graduation

Graduated projects should keep public maintainer, license, Code of Conduct,
security, and contact information current. If a graduated project becomes
inactive, unsafe, unreachable, or closed-source, OSL may open a maintenance
review and update public listings.
